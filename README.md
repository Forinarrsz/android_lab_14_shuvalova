# Unscramble Game

Учебный проект по разработке Android-приложения с использованием современных архитектурных компонентов. 

## Описание
Игра-головоломка, где нужно угадывать зашифрованные слова. Основной фокус работы — реализация **Android App Architecture** и использование **ViewModel** для обеспечения стабильности UI при изменениях конфигурации.

## Технологии
*   **Язык**: Kotlin
*   **UI**: Jetpack Compose
*   **Архитектура**: ViewModel, UI State (Unidirectional Data Flow)
*   **Реактивность**: StateFlow
*   **Тестирование**: JUnit 4

## Функциональность
*   Генерация 10 случайных слов из списка [WordsData.kt].
*   Сохранение состояния (счета и слова) при повороте экрана.
*   Валидация ответов и визуальная индикация ошибок.
*   Диалог завершения игры с возможностью перезапуска.

## Ответы на контрольные вопросы

### 1. Что такое ViewModel и зачем он нужен?
**ViewModel** — это компонент архитектуры, который хранит и управляет данными UI. Без него данные (например, `score`) теряются при повороте экрана (configuration change), так как Activity уничтожается и создается заново. ViewModel «переживает» пересоздание Activity, сохраняя данные в памяти.

### 2. В чём разница между MutableStateFlow и StateFlow?
*   **MutableStateFlow**: Изменяемый поток данных. Используется внутри ViewModel (`_uiState`) для обновления состояния.
*   **StateFlow**: Поток «только для чтения». Экспонируется наружу для UI (`uiState`), чтобы интерфейс мог наблюдать за данными, но не мог их менять напрямую.
*   **Backing property pattern**: Практика использования приватной изменяемой переменной и публичной неизменяемой для защиты целостности данных.

### 3. Почему в ViewModel нельзя хранить Context или View?
Это приводит к **утечкам памяти**. Activity может быть уничтожена, но если ViewModel (которая живет дольше) держит на неё ссылку через Context, сборщик мусора не сможет её удалить.

### 4. Чем StateFlow отличается от remember и rememberSaveable?
*   **remember**: Хранит данные только пока Composable находится в композиции. Теряет всё при повороте экрана.
*   **rememberSaveable**: Сохраняет данные в `Bundle`. Подходит для простых типов (Int, String), но не для сложной бизнес-логики.
*   **StateFlow**: Идеален для UI-логики, отделяет данные от интерфейса и легко тестируется.

### 5. Что такое data class в Kotlin?
Специальный класс для хранения данных.
*   **Методы**: Автоматически генерирует `equals()`, `hashCode()`, `toString()` и `copy()`.
*   **Функция copy()**: Позволяет создать новый объект состояния на основе старого, изменив лишь нужные поля (например, только `score`), что важно для иммутабельного UI State.

### 6. Как работает collectAsState() в Compose?
Это функция-адаптер, которая превращает поток `StateFlow` в состояние, понятное Compose (`State`). Когда значение во ViewModel меняется, Compose видит это и автоматически запускает **рекомпозицию** (перерисовку) нужных частей экрана.

### 7. Зачем разделять код на пакеты data и ui?
Это поддерживает принцип **разделения ответственности** (Separation of Concerns). Пакет `data` отвечает за то, *что* мы отображаем, а `ui` — за то, *как* мы это отображаем. Это делает код масштабируемым и тестируемым.

### 8. Что происходит с ViewModel при process death?
При полной смерти процесса (когда Android убивает приложение из-за нехватки памяти) данные во ViewModel **теряются**. Для долгосрочного хранения нужно использовать **Database (Room)** или **DataStore**.
